## 1. Project Overview

**LockIn** is a gamified daily task tracker where the user's lock screen wallpaper progressively reveals itself as daily tasks are completed. The core loop:

1. User defines daily tasks (title, icon, priority).
2. Tasks appear as icons on the lock screen widget.
3. Each completion re-renders the wallpaper with a higher fill percentage.
4. 100% completion = fully revealed wallpaper + streak increment + optional reward.
5. Midnight resets the day; streak tracks consecutive full-completion days.

**Platform targets:** iOS 16+ and Android 12+ (API 31+).  
**Framework:** React Native via Expo (managed + bare hybrid workflow).

## 4. Module Breakdown

### `app/` — Routing

Uses **Expo Router v3** (file-based). Every file in `app/` is a route. Modals use the `(modal)` group pattern with `presentation: 'modal'` in the layout. Tabs are defined in `app/(tabs)/_layout.tsx`.

No business logic lives in route files. Route files import components and call hooks — nothing else.

### `src/components/` — UI Components

Three categories:

- **Feature components** (`tasks/`, `wallpaper/`, `gamification/`) — tied to a domain, import from stores.
- **UI primitives** (`ui/`) — stateless, take all data via props, no store imports.
- All components are named exports. No default exports for components.

### `src/stores/` — Zustand Stores

One store per domain. Each store file exports:
- The store hook (`useTaskStore`)
- Typed selectors (e.g., `selectCompletedTasks`)
- The store's action types as a TypeScript interface

Stores do **not** call native APIs directly — they call services.

### `src/engine/` — Wallpaper Rendering Pipeline

The most performance-critical module. Runs on a **Reanimated worklet** thread, never the JS thread. `WallpaperRenderer.ts` is the single entry point called by `WallpaperSyncService`. It accepts a `RenderJob` and resolves a `file://` URI to the rendered PNG.

### `src/native/` — Native Bridges

TypeScript wrappers over the local Expo modules in `/modules/`. Abstracts platform differences. All other modules import from here, never from `/modules/` directly.

    ↕  hydrate on launch / persist on write

  uncompleteTask: (taskId: string) => Promise<void>;

`completeTask` is the hottest path in the app. Its sequence:

4. Recalculate `completionPct`.
5. Trigger `WallpaperSyncService.scheduleRender()`.
6. Fire haptic + sound if enabled.
7. Check if streak should increment.

## 7. Wallpaper Engine

### Render Pipeline

```
completeTask()
    → WallpaperSyncService.scheduleRender(completionPct)
        → WallpaperRenderer.render(job: RenderJob)
            → ImageCompositor.composite(base, color, mask)
                ← MaskGenerator.generate(revealMode, pct)
            → write PNG to file:// cache
        → WallpaperManager.setWallpaper(uri)  [native]
        → update DailyLog.renderedWallpaperUri
```

### RenderJob

```typescript
interface RenderJob {
  wallpaper: Wallpaper;
  completionPct: number;        // 0–100
  outputSize: { width: number; height: number }; // device screen resolution
}
```

### MaskGenerator

Generates an `SkPath` (Skia path) that defines the revealed region at a given percentage:

- **`radial`** — circular reveal from center, radius grows with `pct`.
- **`wipe`** — left-to-right linear wipe.
- **`scatter`** — random cells (seeded by date) fill in as `pct` increases. Same seed = same cell order every day for the same wallpaper.

### ImageCompositor

Uses Skia's `Paint` with `BlendMode.srcIn` to apply the mask over the color layer, composited on top of the grayscale base:

```
Layer 0: grayscale(sourceImage)        — always visible
Layer 1: colorImage CLIP mask(pct)     — revealed region only
```

The render runs inside a `runOnJS`-free worklet. Output is written to `FileSystem.cacheDirectory/wallpapers/{date}-{pct}.png`. Old renders for the same date are deleted before writing.

### Performance Budget

- Render must complete in < 300ms on a mid-range device.
- Output PNG is capped at device native resolution (no upscaling).
- Re-renders are **debounced 500ms** — rapid task taps queue one render, not N.

## 8. Lock Screen Integration

### iOS

| Feature | API | Constraint |
|---|---|---|
| Lock screen widget | WidgetKit (Swift) | Static snapshot; updates on timeline |
| Widget task icons | SwiftUI `HStack` of SF Symbols | Max 8 icons comfortably |
| Completion state | App Group MMKV → widget reads `today_state.json` | Widget cannot call JS |
| Wallpaper | **Cannot set programmatically** | User applies via Shortcuts automation |
| Live Activities | ActivityKit | Shows progress bar on Dynamic Island |

**iOS Wallpaper Workaround:** On 100% completion, the app deep-links into Shortcuts via `shortcuts://run-shortcut?name=LockInApply`. The user sets this up once during onboarding. The Shortcut runs a "Set Wallpaper" action using the rendered PNG shared via Files app.

**Widget data flow (iOS):**

```
App writes → App Group container (group.com.lockin.shared)
    → today_state.json  { tasks, completedIds, pct, wallpaperSnapshotUri }
WidgetKit reads → renders SwiftUI timeline entry
Widget reloads → WidgetCenter.shared.reloadTimelines(ofKind: "LockInWidget")
    triggered by: task complete, app foreground, background fetch
```

### Android

| Feature | API | Constraint |
|---|---|---|
| Lock screen widget | AppWidget + KeyguardManager | Requires `BIND_APPWIDGET` permission |
| Wallpaper | `WallpaperManager.getInstance().setBitmap()` | Sets both lock + home or lock only (API 31+) |
| Shared state | ContentProvider or MMKV shared process | Widget reads directly |
| Live progress | Custom RemoteViews with ProgressBar | Updated via `AppWidgetManager` |

**Android wallpaper set is silent and instant** — no user interaction required. This is the primary platform for the core experience.

## 9. Gamification System

### Streak Logic

Implemented in `src/services/StreakService.ts`.

```
On each DailyLog created (midnight reset):
  1. Load yesterday's DailyLog
  2. If completionPct === 100 AND yesterday was consecutive:
       streakCount += 1
       longestStreak = max(streakCount, longestStreak)
  3. If completionPct < 100:
       streakCount = 0
  4. Persist UserProfile
```

**Timezone safety:** All date comparisons use the device's local calendar date (`toLocaleDateString()`), never UTC. The "day" boundary is local midnight.

### Unlock Conditions

```typescript
// src/constants/revealModes.ts

export const UNLOCK_CONDITIONS: Record<string, (profile: UserProfile) => boolean> = {
  '3-day-streak':   (p) => p.streakCount >= 3,
  '7-day-streak':   (p) => p.streakCount >= 7,
  '30-day-streak':  (p) => p.streakCount >= 30,
  'first-complete': (p) => p.longestStreak >= 1,
};
```

`StreakService.checkUnlocks()` runs after every streak update and calls `wallpaperStore.unlock(id)` for newly eligible wallpapers.

### Reward Feedback Sequence

On task completion (orchestrated in `useTaskCompletion` hook):

```
1. Optimistic UI update (instant)
2. Haptics.impactAsync(ImpactFeedbackStyle.Medium)  — immediate
3. TaskCompletionButton scale animation (Reanimated spring) — 200ms
4. If 100%: celebration animation + Haptics.notificationAsync(SUCCESS)
5. WallpaperRenderer debounced render — 500ms after last tap
6. If sound enabled: Audio.playAsync(completionSound)
```

## 14. Background Jobs

### Midnight Reset

Registered via `expo-background-fetch` with a task named `'LOCKIN_DAILY_RESET'`.

Sequence:
1. Check if `todayDate !== storedCurrentDate`.
2. Archive yesterday's DailyLog (mark complete/incomplete).
3. Run `StreakService.evaluate()`.
4. Run `StreakService.checkUnlocks()`.
5. Create new DailyLog for today with recurring tasks.
6. Render 0% wallpaper for today.
7. Update MMKV shared state.
8. Reload widget.

**Fallback:** On every app foreground event (`AppState` change to `'active'`), `useDailyReset` hook checks if the reset has run today. If not, it runs synchronously. This guarantees correctness even when iOS kills the background task.

### Deferred Wallpaper Render

`WallpaperSyncService` uses a debounce queue. Renders are never triggered more than once per 500ms. If the app is backgrounded mid-render, the render is queued for the next foreground event.

## 15. Platform Differences

| Capability | iOS | Android |
|---|---|---|
| Set wallpaper silently | ❌ Requires Shortcuts workaround | ✅ WallpaperManager API |
| Lock screen widget | ✅ WidgetKit (SwiftUI) | ✅ AppWidget (RemoteViews) |
| Live progress indicator | ✅ Live Activities (Dynamic Island) | ✅ Persistent notification |

| Widget refresh rate | System-throttled, ~15min minimum | On-demand via broadcast |

**Feature flags** in `src/constants/features.ts`:

```typescript
export const FEATURES = {
  CLOUD_SYNC:    process.env.EXPO_PUBLIC_ENABLE_CLOUD_SYNC === 'true',
  LIVE_ACTIVITIES: Platform.OS === 'ios',
  SILENT_WALLPAPER: Platform.OS === 'android',
};
```

## 18. Key Conventions

### Naming

- **Files:** `PascalCase` for components, `camelCase` for everything else.
- **Hooks:** always prefixed `use` — `useTaskCompletion`, `useDailyReset`.
- **Services:** suffixed `Service` — `StreakService`, `WallpaperSyncService`.
- **Stores:** suffixed `Store` — `taskStore`, `dailyLogStore`.
- **Types/Interfaces:** suffixed `.types.ts`, interface names `PascalCase`.

### Component Rules

- No default exports for components. Named exports only.
- Props interfaces defined in the same file, named `{ComponentName}Props`.
- No business logic in components. Logic lives in hooks or services.
- Components never import from `src/db/` directly — only from stores or hooks.

### Import Order (enforced by ESLint)

```
1. React
2. React Native / Expo
3. Third-party libraries
4. Internal absolute imports (src/...)
5. Relative imports
6. Types (import type)
```

### Error Handling

- All async operations in services use `Result<T, E>` pattern (no thrown errors crossing module boundaries).
- Native module calls are wrapped in try/catch in `src/native/` — never propagate native exceptions into JS.
- User-facing errors are surfaced via a global `useToast()` hook, not console.error.

### Git Conventions

- Branch: `feature/`, `fix/`, `chore/` prefixes.
- Commits: Conventional Commits (`feat:`, `fix:`, `chore:`, `perf:`).
- No direct commits to `main`. PRs require passing CI (lint + unit tests).
